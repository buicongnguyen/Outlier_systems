1. Algorithm classics

Q1. Two Sum

easyhash map

The "hello world" of coding interviews — first problem in Blind 75; still used as a warm-up screen everywhere.

Given an array of integers nums and an integer target, return the indices of the two numbers that add up to target. Exactly one solution exists; you may not use the same element twice. Example: nums=[2,7,11,15], target=9 → [0,1].

Approach

One pass with a hash map from value → index. For each number, check whether its complement (target - num) has already been seen. The brute force compares all pairs in O(n²); the map trades O(n) space for one lookup per element.

def two_sum(nums, target):
    seen = {}                        # value -> index
    for i, num in enumerate(nums):
        want = target - num
        if want in seen:
            return [seen[want], i]
        seen[num] = i                # store AFTER the check: no self-pairing
O(n) time, O(n) space

Pitfalls & follow-ups

Insert into the map after checking, or target = 2×num matches the element with itself. Follow-ups: input sorted (two pointers, O(1) space), find all pairs, or 3Sum below.

Q2. Find the Extra Character

easybit tricks / counting

Reported on Glassdoor for Google software-engineer screens: two strings with the same characters, one has a single extra.

String t is s shuffled plus one extra character. Return that character. Example: s="abcd", t="cbdae" → "e".

Approach

XOR every character code of both strings together: pairs cancel (x⊕x=0), leaving exactly the extra character. Equivalent one-liner: difference of summed character codes. Both beat the obvious Counter-diff on space.

def extra_char(s, t):
    acc = 0
    for ch in s + t:
        acc ^= ord(ch)      # duplicates cancel out pairwise
    return chr(acc)

# or: chr(sum(map(ord, t)) - sum(map(ord, s)))
O(n) time, O(1) space

Pitfalls & follow-ups

State the invariant (XOR is commutative, associative, self-inverse) — that's what is being tested. Follow-up: the same trick solves "single number" (every element appears twice except one) and "missing number in 1..n".

Q3. 3Sum — Unique Triplets Summing to Zero

mediumsorting + two pointers

Reported on Glassdoor for Meta/Google onsites; a Blind 75 staple with a famous duplicate-handling trap.

Given an integer array, return all unique triplets [a,b,c] with a+b+c=0. Example: [-1,0,1,2,-1,-4] → [[-1,-1,2],[-1,0,1]].

Approach

Sort, then for each anchor index run two pointers over the remainder looking for -anchor. Sorting makes duplicate-skipping possible: advance past equal values at the anchor and at both pointers after each hit.

def three_sum(nums):
    nums.sort()
    res = []
    for i in range(len(nums) - 2):
        if i and nums[i] == nums[i-1]:          # skip duplicate anchors
            continue
        if nums[i] > 0: break                    # sorted: no zero-sum ahead
        lo, hi = i + 1, len(nums) - 1
        while lo < hi:
            s = nums[i] + nums[lo] + nums[hi]
            if s < 0:   lo += 1
            elif s > 0: hi -= 1
            else:
                res.append([nums[i], nums[lo], nums[hi]])
                while lo < hi and nums[lo] == nums[lo+1]: lo += 1
                while lo < hi and nums[hi] == nums[hi-1]: hi -= 1
                lo += 1; hi -= 1
    return res
O(n²) time, O(1) extra space (beyond sort/output)

Pitfalls & follow-ups

The interview is really about the three duplicate-skips — a set-of-tuples dedupe works but signals brute force. Follow-ups: 3Sum-closest, 4Sum (one more anchor loop), k-Sum recursion.

Q4. Merge Overlapping Intervals

mediumsorting / sweep

Blind 75 interval category; constantly reported across Glassdoor for calendar/booking-flavored screens.

Given intervals [[1,3],[2,6],[8,10],[15,18]], merge all overlapping ones → [[1,6],[8,10],[15,18]].

Approach

Sort by start. Sweep left to right keeping the last merged interval: if the next start fits inside it (start ≤ last_end), extend the end with max; otherwise emit a new interval. Sorting is what turns "compare everything with everything" into one linear pass.

def merge(intervals):
    intervals.sort(key=lambda iv: iv[0])
    out = []
    for start, end in intervals:
        if out and start <= out[-1][1]:          # overlaps (or touches) last
            out[-1][1] = max(out[-1][1], end)    # extend — max matters!
        else:
            out.append([start, end])
    return out
O(n log n) time, O(n) output

Pitfalls & follow-ups

The max() is the classic bug: [1,10],[2,3] must stay [1,10]. Clarify whether touching intervals ([1,2],[2,3]) merge. Follow-ups: insert-interval into a sorted list, minimum meeting rooms (heap of end times).

Q5. Validate a Binary Search Tree

mediumtrees / recursion

Reported on Glassdoor for Google ("decide if the given binary tree is a BST"); a canonical recursion-with-invariants test.

Return true iff the tree is a valid BST: every node in a left subtree is less than its ancestor node, every node in a right subtree greater — not just the immediate children.

Approach

Pass down the open interval each node must fall in: root gets (-∞, +∞); going left tightens the upper bound, going right the lower. Alternative with the same power: an in-order traversal of a BST must be strictly increasing.

def is_bst(node, lo=float("-inf"), hi=float("inf")):
    if node is None:
        return True
    if not (lo < node.val < hi):        # must respect ALL ancestors
        return False
    return (is_bst(node.left,  lo, node.val) and
            is_bst(node.right, node.val, hi))
O(n) time, O(h) recursion stack

Pitfalls & follow-ups

The famous wrong answer checks only left.val < node.val < right.val — it accepts a grandchild that violates a grandparent's bound. Also decide how duplicates are treated (strict < here). Follow-ups: recover a BST with two swapped nodes; kth smallest via in-order.

Q6. Longest Increasing Subsequence

mediumDP / binary search

Reported on Glassdoor with the explicit ask to explain BOTH the O(n²) and O(n log n) solutions.

Length of the longest strictly increasing subsequence (not necessarily contiguous). Example: [10,9,2,5,3,7,101,18] → 4 (2,3,7,101).

Approach

DP: dp[i] = best LIS ending at i, looking back at all j<i — O(n²). Patience method: keep tails[k] = smallest possible tail of an increasing subsequence of length k+1; each element binary-searches its slot. tails stays sorted, its length is the answer.

def lis_quadratic(nums):                 # the explainable one
    dp = [1] * len(nums)
    for i in range(len(nums)):
        for j in range(i):
            if nums[j] < nums[i]:
                dp[i] = max(dp[i], dp[j] + 1)
    return max(dp, default=0)

import bisect
def lis_nlogn(nums):                     # the impressive one
    tails = []                           # tails[k] = min tail of LIS len k+1
    for x in nums:
        k = bisect.bisect_left(tails, x) # first tail >= x
        if k == len(tails): tails.append(x)
        else:               tails[k] = x # smaller tail = more room to grow
    return len(tails)
O(n²) / O(n log n) time

Pitfalls & follow-ups

tails is NOT the subsequence itself — only its length is meaningful (reconstructing needs parent links). bisect_left vs bisect_right is the strict-vs-non-strict switch. Follow-up: Russian doll envelopes = LIS after a sort trick.

Q7. Top K Frequent Elements

mediumhash map + heap

Blind 75 heap category; the "log analysis" screen question in countless Glassdoor reports.

Given an array and k, return the k most frequent elements. Example: [1,1,1,2,2,3], k=2 → [1,2].

Approach

Count with a hash map, then select. A size-k min-heap over (count, value) gives O(n log k) — the heap holds the current best k, and anything smaller than its root can't be top-k. Bucket sort by count reaches O(n): index buckets by frequency (at most n), walk from the top.

from collections import Counter
import heapq

def top_k(nums, k):
    counts = Counter(nums)                          # O(n)
    return [v for v, _ in
            heapq.nlargest(k, counts.items(), key=lambda kv: kv[1])]

def top_k_bucket(nums, k):                          # O(n) version
    counts = Counter(nums)
    buckets = [[] for _ in range(len(nums) + 1)]    # index = frequency
    for val, c in counts.items():
        buckets[c].append(val)
    out = []
    for c in range(len(buckets) - 1, 0, -1):
        for val in buckets[c]:
            out.append(val)
            if len(out) == k: return out
O(n log k) heap / O(n) bucket

Pitfalls & follow-ups

Say why it beats full sorting (O(n log n) wastes work ordering elements you'll discard). Follow-ups: top-k in a stream (keep the k-heap), across sharded logs (local top-k, then merge — connects to the distributed section of this hub).

Q8. LRU Cache with O(1) Operations

mediumhash map + doubly linked list

Possibly the most-asked design-coding question anywhere (LeetCode 146); reported at Twitch, Meta, Amazon and dozens more on Glassdoor.

Design a fixed-capacity cache: get(key) returns the value or -1; put(key, value) inserts/updates and evicts the least recently used entry when full. Both must run in O(1) average time.

Approach

Recency order needs a sequence with O(1) move-to-front and remove-tail: a doubly linked list. Key lookup needs O(1): a hash map pointing at the list nodes. Every get/put unlinks the node and re-links it at the front; eviction pops the tail. Python's OrderedDict packages exactly this pair.

from collections import OrderedDict

class LRUCache:
    def __init__(self, capacity):
        self.cap = capacity
        self.od = OrderedDict()          # front = oldest, end = newest

    def get(self, key):
        if key not in self.od: return -1
        self.od.move_to_end(key)         # O(1): relink node as most recent
        return self.od[key]

    def put(self, key, value):
        if key in self.od:
            self.od.move_to_end(key)
        self.od[key] = value
        if len(self.od) > self.cap:
            self.od.popitem(last=False)  # O(1): evict least recent
O(1) get / put, O(capacity) space

Pitfalls & follow-ups

If asked to build it raw: sentinel head/tail nodes remove all the null-pointer edge cases; a plain singly linked list fails (O(n) unlink). Follow-ups: thread-safety (one lock, or sharded locks), TTL entries, LFU (LeetCode 460 — count buckets), and "what does Redis actually do?" (approximate LRU sampling).

Q9. Bounded Blocking Queue (Producer–Consumer)

mediumconcurrency / condition variables

The staple concurrency interview (GeeksforGeeks, Hello Interview LLD); tests mutexes, condition variables, and spurious wakeups in one problem.

Implement a fixed-capacity queue for multiple producer and consumer threads: put() blocks while full, take() blocks while empty. No busy-waiting.

Approach

One mutex guards the buffer; two condition variables carry the two wait reasons (not_full for producers, not_empty for consumers). Waits sit in while loops re-checking the predicate — wakeups are hints, not guarantees — and each side signals the other side after mutating.

import threading
from collections import deque

class BoundedQueue:
    def __init__(self, capacity):
        self.buf = deque()
        self.cap = capacity
        self.lock = threading.Lock()
        self.not_full  = threading.Condition(self.lock)
        self.not_empty = threading.Condition(self.lock)

    def put(self, item):
        with self.not_full:
            while len(self.buf) == self.cap:   # WHILE, never if:
                self.not_full.wait()           # spurious wakeups + races
            self.buf.append(item)
            self.not_empty.notify()            # wake one waiting consumer

    def take(self):
        with self.not_empty:
            while not self.buf:
                self.not_empty.wait()
            item = self.buf.popleft()
            self.not_full.notify()              # wake one waiting producer
            return item
O(1) per operation; waiting threads sleep (no CPU burn)

Pitfalls & follow-ups

The three classic mistakes: if instead of while around wait; one shared condition with notify() (a producer can wake a producer — use two conditions or notify_all); signaling before releasing consistency. Follow-ups: implement with two semaphores (empty/full slots), poison-pill shutdown, and why unbounded queues break backpressure (ties to the distributed section).

Q10. Thread-Safe Token-Bucket Rate Limiter

mediumconcurrency / clocks

Standard at infra teams (AlgoMaster, Hello Interview, system-design blogs): "allow N requests/sec with bursts — implement it."

Implement allow() → bool: requests consume a token; tokens refill at rate per second up to capacity (burst size). Must be correct under concurrent callers.

Approach

Lazy refill: no background thread — on each call, compute tokens earned since the last refill from a monotonic clock, cap at capacity, then atomically check-and-decrement under a lock. The check and the decrement must be one critical section or two threads both pass on the last token.

import threading, time

class TokenBucket:
    def __init__(self, rate, capacity):
        self.rate = rate                  # tokens added per second
        self.cap = capacity               # max burst
        self.tokens = capacity
        self.last = time.monotonic()      # NEVER time.time(): it can jump
        self.lock = threading.Lock()

    def allow(self):
        with self.lock:                   # read-refill-decrement is atomic
            now = time.monotonic()
            self.tokens = min(self.cap,
                              self.tokens + (now - self.last) * self.rate)
            self.last = now
            if self.tokens >= 1:
                self.tokens -= 1
                return True
            return False                  # caller: reject or queue
O(1) per call, O(1) state per bucket

Pitfalls & follow-ups

Wall-clock time breaks on NTP jumps — use monotonic. Fractional tokens are fine (float); per-key buckets go in a dict (mind cleanup). Follow-ups: token bucket vs leaky bucket vs sliding window; distributed version (state in Redis, refill+decrement in one atomic Lua script — the same race, solved the same way).

Q11. Retry with Exponential Backoff and Jitter

easyreliability patterns

Asked as "make this API call robust" at AWS-style shops — the AWS Builders' Library pattern as a coding exercise.

Wrap a flaky function so it retries transient failures up to max_attempts times, waiting exponentially longer between tries, with randomized jitter, and a cap on the delay.

Approach

Delay for attempt n is random.uniform(0, min(cap, base × 2ⁿ)) — "full jitter". The exponential spreads load over time; the randomization decorrelates clients so a synchronized failure doesn't retry as a synchronized stampede.

import random, time

def retry(fn, max_attempts=5, base=0.1, cap=10.0,
          retryable=(TimeoutError, ConnectionError)):
    for attempt in range(max_attempts):
        try:
            return fn()
        except retryable:
            if attempt == max_attempts - 1:
                raise                          # budget exhausted: surface it
            delay = random.uniform(0, min(cap, base * 2 ** attempt))
            time.sleep(delay)                  # full jitter: U(0, exp cap)
Worst-case wait bounded by Σ min(cap, base·2ⁿ)

Pitfalls & follow-ups

Only retry transient errors (a 400 will fail forever; a timeout may have succeeded server-side — so the operation must be idempotent). Never retry without jitter. Follow-ups: honor an overall deadline instead of attempt count, retry budgets (retries as % of traffic), and where the circuit breaker takes over — this is the coding half of the retry-amplification postmortem in the distributed section.

Q12. Consistent Hashing Ring

mediumdistributed systems / binary search

The "design a cache cluster's key placement" question — asked wherever sharding is discussed; the algorithm inside Dynamo, Cassandra, and memcached clients.

Implement add_node, remove_node, and get_node(key) such that keys spread evenly and adding/removing a node remaps only ~1/N of keys (naive hash(key) % N remaps almost everything).

Approach

Hash nodes onto a circle (0..2³²); a key belongs to the first node clockwise from its own hash — a binary search over the sorted node-hash list. Each physical node gets many virtual nodes (replicas of its hash with different salts) so load spreads evenly and a leaving node's keys scatter to everyone, not one neighbor.

import bisect, hashlib

def _h(s):
    return int(hashlib.md5(s.encode()).hexdigest(), 16) % (2 ** 32)

class HashRing:
    def __init__(self, vnodes=100):
        self.vnodes = vnodes
        self.ring = []            # sorted list of (hash, node)

    def add_node(self, node):
        for i in range(self.vnodes):
            bisect.insort(self.ring, (_h(f"{node}#{i}"), node))

    def remove_node(self, node):
        self.ring = [(h, n) for h, n in self.ring if n != node]

    def get_node(self, key):
        if not self.ring: return None
        i = bisect.bisect(self.ring, (_h(key), chr(0x10FFFF)))
        return self.ring[i % len(self.ring)][1]   # wrap = the "circle"
O(log V) lookup, O(V log V) add — V = total virtual nodes

Pitfalls & follow-ups

Without virtual nodes, two physical nodes split the circle terribly unevenly, and removal dumps an entire arc on one survivor. The modulo wrap at the end IS the ring. Follow-ups: weighted nodes (more vnodes = more traffic), how this connects to the shard-key discussion in the databases section, and bounded-load consistent hashing.
