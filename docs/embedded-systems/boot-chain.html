Boot is a sequence of progressively richer execution environments

Each stage runs with more memory and device support than the stage before it. Secure designs also make each handoff a trust decision: authenticate the next image, validate its policy, then transfer control.

1. Staged boot sequence

  1. Reset and immutable Boot ROM

    CPU state is minimal. ROM samples boot straps or fuses, initializes only what is necessary to read a boot device, and selects a recovery or normal path.

  2. Authenticate the first mutable stage

    ROM hashes and verifies a signed header or manifest against a key anchored in ROM or one-time-programmable storage. Anti-rollback policy may reject an old but correctly signed image.

  3. Small first-stage loader

    Code-size and SRAM constraints dominate. It initializes clocks and DRAM, then loads a richer firmware stage. U-Boot systems may call these TPL or SPL; names vary.

  4. Platform firmware / bootloader

    Discover boot media, initialize consoles and buses, apply update/recovery policy, select an OS image and assemble kernel parameters.

  5. Load the OS payload

    Place the kernel, optional initramfs and hardware description at valid memory addresses; verify required signatures; prepare the architecture-specific entry contract.

  6. Transfer control to the kernel

    Pass the device tree or ACPI/UEFI information, command line and memory map. Firmware boot services end when the OS takes ownership.

  7. Kernel initialization

    Set up architecture state, memory management, scheduler, interrupts, timers, buses and initcalls; probe built-in drivers and prepare the root filesystem.

  8. Start userspace

    Execute PID 1 from initramfs or the real root filesystem. PID 1 starts services, including the NPU runtime daemon if the design uses one.

2. Secure, verified and measured boot

Root of trust

An immutable key, key hash or verification routine anchors the chain. If the anchor can be replaced without authorization, downstream signatures have no value.

Verified boot

Execution is allowed only after signature and policy checks. The chain can cover firmware, kernel, device tree, initramfs and a verified root filesystem.

Measured boot

Stages record hashes into protected measurement registers for later attestation. Measurement reports what ran; it does not automatically prevent execution.

Anti-rollback

Signed old firmware can still contain known vulnerabilities. Monotonic counters or secure version policy reject images below the accepted security version.

Revocation

Compromised signers must be removable through authenticated revocation lists or key rotation without destroying the recovery path.

A/B updates

Write the inactive slot, authenticate it, try it with a bounded boot counter, then mark success from a healthy system. Fall back after repeated failures.

Authentication is not confidentiality

Secure Boot verifies origin and integrity. Encrypting firmware, protecting secret keys and preventing physical extraction require additional controls.

3. UEFI and U-Boot

DimensionUEFIU-Boot
What it isA firmware interface specification plus boot-manager behavior and standard protocols.An open-source bootloader widely used in embedded systems, with board ports, commands and boot methods.
Boot selectionBoot#### options and BootOrder variables identify EFI applications and device paths.Standard boot scans boot devices and boot methods for a usable bootflow; scripts and direct boot commands also exist.
OS handoffLaunches an EFI OS loader or EFI-stub kernel; the OS calls ExitBootServices before taking full ownership.Can load a kernel, initrd and DTB directly with bootm/booti/bootz, or expose UEFI services and run an EFI application.
Hardware descriptionCommonly ACPI on PCs/servers; AArch64 embedded platforms may also pass a device tree.Strong device-tree integration for both U-Boot's own driver model and the Linux payload.
Secure bootDefines image validation and key databases such as PK, KEK, db and dbx.Supports signed FIT configurations, verified-boot chains and platform-specific key storage.

They can coexist

U-Boot can implement UEFI protocols and launch GRUB or an EFI-stub Linux kernel. “UEFI versus U-Boot” is not always an exclusive platform choice.

4. Device tree: describing non-discoverable hardware

A DTB is a compact tree of nodes and properties. It does not contain drivers; it describes hardware so the kernel can instantiate devices, match drivers and supply resources. 

npu@4a000000 {
    compatible = "vendor,npu-v2", "vendor,npu";
    reg = <0x0 0x4a000000 0x0 0x10000>;
    interrupts = <0 92 4>;
    clocks = <&accel_clk>;
    resets = <&resetc 7>;
    iommus = <&smmu 0x34>;
    memory-region = <&npu_fw>, <&npu_shmem>;
    firmware-name = "vendor/npu-fw.bin";
    status = "okay";
};

compatible

Ordered strings describe the programming model from most specific to fallback. Drivers match an OF table against these values.

reg

Address and size tuples identify MMIO or bus resources. Cell counts come from the parent bus.

interrupts

Interrupt specifiers are interpreted by the referenced interrupt domain. The raw numbers are not portable without the binding.

clocks and resets

Provider references let the driver request correct sequencing through kernel frameworks instead of hard-coding registers.

iommus

Associates the device with an IOMMU and stream identifier used to isolate DMA transactions.

reserved-memory

Excludes firmware/shared regions from normal allocation; consumers reference the regions with memory-region.

This node is illustrative, not a binding

Real property names, cell encodings and required resources come from the device's reviewed binding and SoC integration documentation.

5. Linux kernel initialization

  1. Early architectureEstablish CPU mode, early console, boot parameters and physical memory view.
  2. MemoryBuild page tables, allocators and reserved regions; preserve firmware/DTB/initramfs memory.
  3. Core kernelInitialize scheduler, timekeeping, interrupts, workqueues and synchronization infrastructure.
  4. Subsystem initRun ordered initcalls for buses, filesystems, security and built-in drivers.
  5. Device probeMatch devices to drivers; defer probes whose clocks, regulators, IOMMUs or firmware are not ready.
  6. Root filesystemUse initramfs to discover storage/unlock volumes if needed, then mount or switch to the real root.
  7. PID 1Execute the first userspace program; a missing or invalid init causes a kernel panic.

Boot failure gates

No ROM output

Confirm reset, power rails, strap/fuse boot selection and whether ROM exposes any observable UART, USB or status code.

Signature rejection

Check image format, signed bytes, key hierarchy, revocation database, security version and board lifecycle state.

Bootloader runs, kernel silent

Validate load/entry address, architecture mode, DTB pointer, console command line, memory overlap and kernel image type.

Kernel cannot find device

Inspect firmware description, status property, compatible match, resource providers, deferred probe messages and driver configuration.

Cannot mount root

Check root= arguments, storage driver availability, initramfs contents, filesystem support and encryption/RAID dependencies.

PID 1 failure

Verify executable path, permissions, interpreter, shared libraries, architecture and the initramfs-to-root switch.
