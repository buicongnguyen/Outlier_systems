System map

The host owns policy, memory protection and application lifecycle. The NPU firmware owns the accelerator's local scheduling and hardware programming. A driver-defined control and data path is the contract between them.

Golden path: cold boot to first inference

Treat each arrow as a contract. If a stage fails, debug the earliest invariant that is not yet true.

  1. Boot ROMSelect boot source and authenticate the first mutable stage.
  2. Secure bootExtend trust through firmware, configuration and recovery policy.
  3. BootloaderInitialize required hardware and select the OS boot flow.
  4. Host kernelInitialize memory, interrupts, buses, drivers and userspace.
  5. Device discoveryEnumerate PCIe or instantiate a firmware-described platform device.
  6. NPU driverClaim resources, map registers, enable power and establish isolation.
  7. NPU firmwareAuthenticate, transfer, start and negotiate the control ABI.
  8. DMA + IOMMUCreate constrained device-visible mappings for shared buffers.
  9. RuntimeExpose contexts, queues, synchronization and compiler services.
  10. Model loadingCompile or deserialize graphs, allocate tensors and upload constants.
  11. InferenceSubmit commands, execute, signal completion and return outputs.

Two hardware variants to keep separate

SoC-integrated NPU

Firmware tables usually describe register ranges, clocks, resets, interrupts, IOMMU links and reserved memory. A Linux platform driver may use remoteproc-style lifecycle control or a vendor subsystem to load firmware into tightly coupled memory.

Discrete PCIe NPU

The PCIe link trains, configuration space is enumerated, BARs and interrupt vectors are assigned, and a PCI driver binds by vendor/device ID. The device may boot internal ROM before accepting host- supplied firmware through BAR, DMA or a management interface.

Do not memorize one vendor's exact boot order as universal

The invariant chain is portable, but ownership differs. Some NPUs authenticate firmware internally; others rely on trusted host software. Some share coherent system memory; others require explicit copies.

How to use this branch

1 · Read for invariants

At each stage, identify who runs, what it trusts, what resources it owns and what evidence proves success.

2 · Draw the handoff

Reproduce the sequence without notes, including error and recovery signals rather than only the happy path.

3 · Take the checkpoint

Answer all five MCQs on each topic page and read every explanation, including the ones you answered correctly.

4 · Diagnose backwards

Start from the failed observable, then walk backward to the earliest missing invariant instead of guessing components.
