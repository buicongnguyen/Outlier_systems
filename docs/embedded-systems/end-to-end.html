A concrete teaching system

Eleven gates from reset to a trusted inference result

This case study uses an ARM64 Linux host with an integrated NPU whose control processor runs a small RTOS. The exact registers are fictional; the ownership, trust, memory and recovery invariants are realistic.

Case-study policy choices

Boot ROM anchors the host chain. U-Boot verifies a signed FIT containing kernel, DTB and initramfs. The host driver accepts NPU firmware only when its signed manifest chains to an accelerator-update key. These are deliberate system policies—not automatic properties of Linux or every NPU.

The complete sequence

  1. Boot ROMAuthenticate SPL from the selected boot slot.
  2. Secure bootExtend trust and enforce version/recovery policy.
  3. U-BootVerify FIT and pass kernel + DTB + initramfs.
  4. Host kernelInitialize Linux, memory, SMMU, buses and userspace.
  5. Device discoveryInstantiate the NPU from device tree resources.
  6. NPU driverOwn power, reset, MMIO, IRQs and DMA isolation.
  7. NPU firmwareAuthenticate, load, start RTOS and handshake.
  8. DMA/IOMMUCreate protected queue and tensor mappings.
  9. RuntimeCreate a context and compile the graph.
  10. Model loadingUpload constants and prepare executable state.
  11. InferenceSubmit, execute, complete and validate output.

Stage-by-stage contracts

Gate 1 · Boot ROM

Actor and action

Immutable SoC ROM samples lifecycle fuses and boot straps, selects A or recovery media, reads the SPL manifest and verifies its signature/key policy.

Invariant after handoff

The first mutable code is authorized for this product and security version; SRAM load ranges and entry point are valid.

Evidence

ROM status/failure code, selected slot, measured hash, boot counter and a bounded transition into SPL.

Gate 2 · Secure loader chain

Actor and action

SPL initializes DRAM, verifies U-Boot proper and transfers control. Each signed manifest carries a security version and allowed next-stage key.

Invariant after handoff

DRAM is usable, the richer bootloader is authenticated, and an old vulnerable image cannot bypass the monotonic rollback policy.

Failure boundary

On authentication or DRAM failure, enter a restricted recovery path that accepts only correctly signed update packages.

Gate 3 · U-Boot OS selection

Actor and action

U-Boot selects the candidate slot, verifies the FIT configuration and loads kernel, DTB and initramfs without overlap.

Invariant after handoff

Linux receives an authenticated hardware description, command line, initramfs and correct architecture entry state.

Evidence

FIT configuration name, verified key identifier, image addresses/sizes and boot-attempt state stored for A/B rollback.

Gate 4 · Linux host initialization

Actor and action

Linux establishes page tables, scheduler, interrupts and timekeeping; initializes SMMU, power domains, clocks and the platform bus; then starts PID 1.

Invariant after handoff

Host memory protection works, required resource providers are ready and userspace can start without exposing the NPU yet.

Evidence

Early console, memory/reserved-region map, SMMU probe, initcall timing, rootfs mount and boot-success marker after health checks.

Gate 5 · NPU discovery

npu@4a000000 {
    compatible = "edgevision,npu-a1";
    reg = <0x0 0x4a000000 0x0 0x10000>;
    interrupts = <0 92 4>, <0 93 4>;
    clocks = <&npu_core_clk>, <&npu_bus_clk>;
    resets = <&resetc 7>;
    power-domains = <&pd_npu>;
    iommus = <&smmu 0x34>;
    memory-region = <&npu_fw>, <&npu_crashlog>;
    firmware-name = "edgevision/npu-a1-fw.bin";
};

Invariant

The platform driver has an authoritative description of MMIO, IRQ, power/reset/clock, IOMMU and reserved-memory resources. A successful DT match does not prove the NPU processor is running.

Gate 6 · Driver ownership

  1. Power domain on; clocks prepared

    Do not access registers until the interconnect path is live.

  2. Reserve/map MMIO; keep NPU reset asserted

    Read a hardware ID and capability register to verify the programming interface.

  3. Attach SMMU and set DMA mask

    No workload-facing DMA occurs before the isolation boundary exists.

  4. Request management and completion IRQs

    Keep sources masked until queue memory and handlers are initialized.

  5. Enter LOADING state

    Userspace sees no usable accelerator node until firmware reaches Ready.

Gate 7 · NPU firmware and RTOS

Host driver

  • Request and authenticate image
  • Copy sections to reserved memory
  • Program boot address
  • Release reset
  • Wait for HELLO

Bootstrap

boot mailbox · shared control page · management IRQ

NPU RTOS

  • Early hardware init
  • Queue scheduler task
  • DMA/fault service task
  • Low-priority telemetry
  • Watchdog supervisor

RTOS priority design

Fault containment and time-critical DMA/engine service outrank command admission; telemetry never blocks control work. The watchdog is fed only after critical-task progress is confirmed.

Gate 8 · DMA/IOMMU and queues

  1. Allocate coherent submission/completion rings

    Store monotonic producer/consumer counters, protocol version and boot generation in a shared queue table.

  2. Map tensor buffers with least privilege

    Inputs are device-readable, outputs are device-writable, and unrelated host pages are absent from the NPU domain.

  3. Publish queue-table IOVA

    A barrier precedes the bootstrap mailbox or doorbell that exposes the address to firmware.

  4. Enable interrupts after state is valid

    Clear stale status first, then unmask management, completion and fault sources.

Gate 9 · Userspace runtime

Create context

The runtime opens the device, negotiates kernel ABI/capabilities and obtains a context isolated from other processes.

Compile graph

Supported operators are partitioned for the NPU, tensor layouts and quantization are selected, and fallback policy is explicit.

Allocate/import buffers

Camera input, host staging and output buffers are imported with fences and appropriate IOMMU permissions.

Gate 10 · Model loading

  1. Validate artifactCheck compiler/runtime/firmware compatibility and model signature policy.
  2. Allocate stateReserve device-local and shared memory with alignment and size bounds.
  3. Upload constantsTransfer weights and microcode; verify checksums where the device protocol supports it.
  4. Create executableFirmware returns a generation-scoped handle; stale handles cannot cross reset.

Gate 11 · First inference

  1. Acquire and fence input

    Wait until the camera/producer has finished writing; synchronize ownership for the NPU.

  2. Publish command descriptor

    Reference model handle, input/output IOVAs, dimensions, deadline class and completion fence; then ring the doorbell.

  3. Firmware schedules engines

    The RTOS validates generation/bounds, performs local admission and programs DMA/tensor engines.

  4. Completion is ordered

    Firmware writes status/output visibility, advances the completion ring and raises the completion IRQ.

  5. Host finalizes

    The driver drains completions, signals the fence, releases temporary mappings and reports timing/errors to the runtime.

  6. Application validates output

    Check status, tensor metadata and an application-level plausibility/quality condition before consuming the result.

How the sequence changes for a discrete PCIe NPU

Discovery moves to PCIe

Link training and configuration-space enumeration replace DT platform-device creation; a DT/ACPI host bridge still describes the root complex.

Resources become BARs/vectors

BAR assignment and MSI/MSI-X allocation replace fixed SoC MMIO/IRQ descriptions for the NPU function.

Reset semantics change

Use device-specific reset, function-level reset, secondary-bus reset or power cycle according to hardware guarantees and peer impact.

Firmware path may be internal

The device may first boot its own ROM/flash, then accept a host firmware update or runtime image through a management channel.

IOMMU remains central

PCIe Requester IDs identify DMA streams; isolation and mapping lifetime rules remain even when discovery differs.

Surprise removal is possible

Hotplug or link loss requires lifetime-safe teardown, failed fences and no continued MMIO access to the removed function.

Production recovery path

  1. DetectWatchdog expiry, queue timeout, fault IRQ, IOMMU fault or link loss.
  2. FenceBlock new submissions and mark all contexts with the failing generation.
  3. QuiesceStop engines or contain DMA; synchronize interrupts and deferred work.
  4. CapturePreserve firmware log, queue heads, fault registers and model/context identifiers.
  5. Reset + reloadRepeat authenticated firmware startup and capability handshake.
  6. RebuildCreate new mappings, queues, model handles and fences under a new generation.
  7. ReportFail or replay requests according to idempotency and deadline policy; never silently lose work.
