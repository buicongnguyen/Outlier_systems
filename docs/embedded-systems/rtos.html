Bounded response, explicit priorities

An RTOS is about predictability—not maximum throughput

Real-time correctness combines the right result with a timing constraint. The scheduler, interrupt design, critical sections, IPC and memory strategy must make worst-case response time explainable.

DeadlinesPreemptionISR → taskWatchdogs

Learning outcomes

Classify timing

Separate hard, firm and soft deadlines and choose evidence appropriate to each.

Trace scheduling

Explain Ready, Running and Blocked transitions under fixed-priority preemption.

Bound interference

Account for ISRs, critical sections, higher-priority tasks and shared resources.

Design supervision

Use watchdogs and health reports without hiding application deadlocks.

1. Core timing model

Latency is a distribution; real-time engineering focuses on the upper bound. Average response can be excellent while rare interrupt masking, cache misses or lock contention still violate a deadline.

Hard deadline

Missing it is a system failure: motor control, protection trips or safety interlocks. Prove or conservatively bound worst-case behavior.

Firm deadline

A late result has no value, but an occasional miss can be handled—for example, dropping a stale sensor frame.

Soft deadline

Lateness degrades quality rather than correctness: UI refresh, logging or best-effort telemetry.

Jitter

Variation in release, start or completion time. Control loops may care about jitter even when every job technically meets its deadline.

Response-time budget

response = release jitter + blocking + higher-priority interference + execution + preemption/interrupt overhead

2. Scheduler and task states

In a common fixed-priority preemptive RTOS, the highest-priority task that is Ready runs. Equal-priority tasks may time-slice, but time slicing is not a substitute for assigning priorities from deadlines and criticality.

State/eventMeaningEngineering consequence
ReadyEligible to run but waiting for CPU time.A higher-priority Ready task normally prevents lower-priority tasks from running.
RunningCurrently executing on a CPU core.It can be preempted, block on an object, delay, or yield.
BlockedWaiting for time, a queue, notification, semaphore or event.Blocking is efficient; spinning wastes the CPU and raises interference.
SuspendedExplicitly removed from scheduling until resumed.Use deliberately; hidden suspend/resume dependencies are difficult to analyze.
ReleaseA timer or event makes a task Ready.If it outranks the running task, a preemptive scheduler switches promptly.
  1. Sensor IRQ fires

    The processor saves minimal context and enters the interrupt handler.

  2. ISR acknowledges and captures

    Clear the source, copy a small sample or descriptor, and avoid blocking APIs.

  3. ISR notifies control task

    An ISR-safe queue, semaphore or direct notification changes the task to Ready.

  4. Scheduler evaluates priorities

    If the awakened task outranks the interrupted task, request a context switch on ISR exit.

  5. Control task runs

    It performs bounded processing, updates outputs, records timing and blocks until the next release.

3. Interrupt design

Keep the top half bounded

Acknowledge the source and move only enough data to prevent loss. Defer parsing, logging, allocation and slow I/O to task context.

Use ISR-safe APIs

RTOS ports usually provide dedicated FromISR-style primitives. They do not block and can report whether a higher-priority task was awakened.

Control interrupt masking

Long global-disable or high-priority critical sections directly increase worst-case interrupt latency. Measure maximum masked time.

Budget nesting

Higher-priority interrupts can preempt lower ones. Include nesting depth, handler time and stack usage in the worst-case design.

Priority numbers are architecture-specific

Some interrupt controllers use a lower numeric value for a higher hardware priority, while many RTOS task APIs use larger numbers for higher task priority. Never infer urgency from the number alone.

4. Priority inversion and bounded sharing

The classic failure requires three priorities and one shared lock.

  1. Low takes a mutex

    A low-priority task begins a short critical section protecting a shared device.

  2. High blocks on the mutex

    The most urgent task cannot proceed even though Low owns the needed resource.

  3. Medium preempts Low

    Unrelated medium-priority work can run indefinitely, indirectly blocking High.

  4. Priority inheritance raises Low

    Low temporarily executes at High's priority, completes the critical section and releases the mutex.

  5. High runs; Low returns to base priority

    The inversion becomes bounded by the protected critical section rather than Medium's workload.

Inheritance is damage control, not permission for large critical sections

Keep lock scope small, avoid blocking while holding a lock, define a global lock order, and prefer single-owner task designs for complex peripherals.

5. Timers, IPC, memory and watchdogs

Timers

Hardware timers drive precise release events; software timers often run callbacks in a shared timer task. A slow callback delays every timer behind it.

IPC

Queues transfer ownership and apply backpressure. Notifications are lightweight one-to-one signals. Semaphores signal availability; mutexes protect ownership.

Memory allocation

Static stacks, fixed-block pools and bounded queues remove fragmentation and unbounded allocator paths. Size stacks from measured high-water marks plus margin.

Watchdogs

A supervisor feeds the hardware watchdog only after critical tasks report progress. Record reset cause and preserve a minimal crash breadcrumb.

Backpressure

Every queue needs a full policy: block with a bound, drop newest, drop oldest, overwrite a sample, or enter a safe state. “The queue never fills” is not a policy.

Tickless idle

Suppress periodic ticks while idle to save power, then program the next wake event. Account for timer drift and wake-up latency.

6. Linux host, RTOS control core and NPU coordination

A heterogeneous system often combines Linux for rich policy and applications, an RTOS for bounded control, and NPU firmware for accelerator-local scheduling. Define ownership before defining messages.

Linux host

  • Security and process isolation
  • Model/runtime lifecycle
  • Large memory and filesystems
  • Updates and diagnostics

Contract

mailbox · shared memory · rpmsg · queue ABI · heartbeat

RTOS / NPU side

  • Bounded control loops
  • Power and safety state
  • Command execution
  • Local watchdog and fault report
Version the ABI

Handshake before commands: protocol version, features, queue shape, address width and cache-coherency model.

Specify reset ownership

Who may reset the remote side, how DMA is quiesced, and what happens to in-flight work?

Use monotonic heartbeats

Distinguish transport silence, a stalled task and an intentionally powered-down subsystem.

Design degraded modes

Define which safety or control functions survive when Linux, the NPU, or the interconnect is unavailable.

Failure patterns to recognize

Missed deadline

Trace release timestamp, ISR delay, Ready-to-Running delay, lock blocking, execution time and completion separately.

Stack overflow

Watch nested interrupts, recursion, large local arrays and library call depth. Enable guards and high-water instrumentation.

Queue saturation

Measure producer/consumer rates, maximum depth and drop counts. Decide whether stale work should be discarded or coalesced.

Watchdog reset loop

Persist reset reason and a boot counter; enter recovery after repeated failures rather than endlessly restarting the same image.
