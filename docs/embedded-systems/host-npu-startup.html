2. Driver probe establishes ownership

  1. Match the device

    The bus finds an ID match and invokes the driver's probe path. A match says the driver understands the programming interface, not that the device is healthy.

  2. Enable resources

    Enable the PCI function or runtime-power domain, acquire regulators/clocks, deassert prerequisite resets and reserve MMIO regions.

  3. Map control registers

    Map BAR or platform MMIO apertures with kernel helpers. Use MMIO accessors and respect posted-write/readback requirements.

  4. Set DMA capabilities

    Negotiate the coherent and streaming DMA mask. Attach the device to the intended IOMMU domain before accepting workloads.

  5. Allocate interrupts

    Request MSI/MSI-X or wired IRQs, establish handlers and keep interrupts masked until queues and status state are valid.

  6. Create software state

    Initialize locks, state machine, reset generation, telemetry and lifetime references before exposing userspace interfaces.

Probe ordering may be deferred

An integrated NPU driver may wait for clocks, regulators, an IOMMU, reserved memory or another firmware provider. Deferred probe is a dependency signal, not automatically a driver bug.

3. Firmware loading and remote start

The Linux firmware API locates image bytes. Device-specific code still controls authentication policy, placement, cache maintenance, reset sequencing, entry point and health monitoring.

  1. Request imageLocate the named firmware from built-in data, cache, filesystem or configured fallback.
  2. Validate containerCheck bounds, architecture, sections, manifest, ABI version and cryptographic policy.
  3. Quiesce NPUHold reset, stop DMA, mask interrupts and make the previous generation inaccessible.
  4. Place firmwareCopy sections to device memory or create DMA mappings the bootstrap engine can read.
  5. Program boot stateSet entry point, page tables, queue bootstrap address, clocks and watchdog policy.
  6. Release resetStart the remote processor and wait for a bounded boot-status transition.
  7. HandshakeExchange magic, ABI version, capabilities, queue counts, address width and coherency flags.
  8. Mark ReadyOnly now expose a usable accelerator context to the runtime.

Host trust does not automatically cover NPU firmware

The NPU image must be included in an authenticated host manifest or verified by an NPU-local immutable root of trust. Otherwise a secure host can still start unauthorized accelerator code with DMA capability.

4. A remote-processor lifecycle model

Linux's remoteproc framework illustrates a useful separation: a common core parses firmware and models lifecycle, while platform operations implement power, load, start, stop, kick and crash handling.

Offline

Clocks may be gated, reset asserted and no DMA permitted. Software retains metadata but not a live queue contract.

Loading

Firmware sections and resource descriptions are validated and placed. Shared memory and virtual devices may be prepared.

Running

The remote processor has acknowledged the ABI and may consume commands. A generation number ties queues to this boot.

Crashed

Stop submissions, capture a coredump/status snapshot, quiesce DMA and decide whether policy allows automatic recovery.

Recovering

Invalidate stale handles, reset hardware, reload firmware and rebuild transport state before clients can resume.

Suspended

Save or discard context explicitly. Firmware availability, IOMMU mappings and queue ownership must be revalidated on resume.

5. Firmware handshake contract

host → NPU: HELLO {
  protocol_major, protocol_minor,
  host_features, queue_table_iova,
  address_bits, cache_coherency,
  boot_generation, nonce
}

NPU → host: READY {
  accepted_protocol, firmware_build,
  npu_features, max_queues,
  max_tensor_bytes, alignment,
  watchdog_period, boot_generation
}
Fail closed on major mismatch

A different major version means layouts or semantics may be incompatible. Do not “try anyway.”

Negotiate optional features

Use intersected capability bits and document fallback behavior for minor-version differences.

Carry boot generation

Reject completions from a previous firmware instance after reset or recovery.

Bound the wait

Every reset, firmware load and Ready wait needs a timeout, diagnostic status and safe recovery transition.
