Safe bytes in, ordered commands, explicit completion

The data path is a distributed protocol across CPU, memory fabric and NPU

Correctness depends on address translation, ownership, cache visibility, queue ordering, interrupt handling and reset generation—not only the command format. The driver turns those rules into a safe API.

Validate at the trust boundary

Treat userspace sizes, offsets, handles and command references as untrusted. Check integer overflow, object lifetime, mapping permissions, queue ownership and firmware-visible bounds before submission.

2. PCIe/SoC resources and MMIO

PCI configuration space

Enumeration reads IDs and capabilities, sizes BARs, assigns addresses and configures link/device features. The kernel PCI core owns this choreography.

BAR or platform MMIO

The driver reserves and maps register apertures, then uses MMIO accessors. Ordinary pointer loads/stores may violate ordering or architecture rules.

Doorbell registers

A write notifies the device that a producer index or queue state changed. Publish descriptors before ringing the doorbell.

Posted writes

Some interconnect writes complete from the CPU's perspective before reaching the device. A defined readback may be needed when ordering is critical.

SoC clocks and resets

Integrated devices may return bus errors or stale values if accessed while clocks are gated or reset remains asserted.

Runtime power

Reference counts and autosuspend must cover every queue, mapping and in-flight command. Resume restores hardware state before access.

3. DMA address domains

A CPU pointer is meaningful inside one process or kernel mapping. An NPU issues bus transactions using a device-visible address. The DMA API bridges those domains and hides platform-specific translation/coherency work.

  1. CPU virtualThe address software dereferences; translated by CPU page tables.
  2. CPU physicalThe backing RAM address managed by the kernel's memory subsystem.
  3. DMA / IOVAThe address programmed into the NPU descriptor or page table.
  4. IOMMU translationMaps the device-visible address to permitted physical pages with access permissions.
Mapping typeTypical useKey rule
Coherent / consistentCommand rings, completion rings, shared control blocks.CPU and device see updates coherently, but ordering barriers can still be required.
StreamingInput/output tensors or one-shot transfer buffers.Map for a direction and lifetime, perform required sync, then unmap after the device is finished.
Imported shared bufferCamera → NPU or NPU → display pipelines.Track attachment, scatter-gather mappings, fences, ownership and per-device cache rules.
Device-local memoryWeights, microcode or scratch optimized for accelerator bandwidth.Host access may require BAR windows or explicit copies; lifetime is tied to device/reset state.

Never unmap a buffer while the NPU can still access it

Completion ordering, cancellation, reset and process teardown must all prove DMA has stopped before mappings and pages are released.

4. IOMMU isolation

Translation

The device issues an I/O virtual address (IOVA); the IOMMU translates it to host physical pages, similar in purpose to CPU page tables.

Containment

Only explicitly mapped pages are reachable. A faulty or hostile command cannot DMA across all RAM if the device and domain are configured correctly.

Per-context isolation

PASID/SVA or separate domains can distinguish processes or guests, but hardware and driver support determine granularity.

Fault reporting

Log stream/PASID, IOVA, access type and reason. Treat repeated faults as a context or device health event, not just a noisy warning.

TLB invalidation

Changing mappings requires invalidating cached translations before pages are reused. Framework APIs enforce the correct ordering.

Not a complete sandbox

MMIO, peer-to-peer paths, shared firmware, side channels and driver vulnerabilities remain. Isolation is a system property.

5. Command and completion queues

A ring buffer is simple only after ownership and memory ordering are explicit.

host:
  slot = submit_ring[producer % ring_size]
  fill_descriptor(slot, opcode, buffers, fence, generation)
  dma_write_barrier()           // descriptor visible before producer
  shared_producer = producer + 1
  mmio_write(DOORBELL, queue_id)

NPU firmware:
  observe producer
  read descriptor
  validate generation + bounds
  execute command
  write completion status
  dma_write_barrier()           // status visible before completion producer
  update completion_producer
  raise interrupt or event
Producer owns free slots

It must never overrun the consumer. Track wrap with monotonic counters or an explicit phase bit.

Consumer owns published slots

It reads a descriptor only after observing the ordered producer update.

Generation rejects stale work

Commands and completions from before reset cannot mutate the new queue instance.

Backpressure is visible

Block, return busy, queue in software or reject according to an explicit latency and memory policy.

6. Interrupts, polling and completions

  1. NPU writes completion

    Firmware records status and output visibility before updating the completion producer.

  2. NPU raises MSI-X/MSI or wired IRQ

    Multiple vectors may separate queues, faults and management events.

  3. Top-half handler acknowledges

    Read status, mask or clear the source as required, record work and wake deferred processing.

  4. Deferred handler drains completions

    Validate queue generation, signal fences, release mappings and wake clients outside hard-interrupt context.

  5. Load policy adapts

    Interrupt coalescing or polling reduces high-rate overhead but trades power and latency; measure the target workload.

Interrupt arrival is not enough

Memory ordering must guarantee the completion record is visible before software treats the interrupt as proof of completion.

7. Reset and recovery protocol

  1. FreezeReject submissions and mark contexts unhealthy.
  2. QuiesceMask interrupts and stop or contain DMA before freeing anything.
  3. CaptureSave fault registers, queue pointers, firmware logs and reset reason.
  4. ResetUse function-level or subsystem reset with bounded completion.
  5. ReloadAuthenticate firmware and repeat the ABI handshake.
  6. RebuildCreate new queues/mappings with a new boot generation.
  7. NotifyFail, replay or resubmit work only according to an idempotency policy.

Data-path failure patterns

IOMMU translation fault

Check IOVA, mapping direction, permissions, device/domain attachment, scatter-gather length and unmap timing.

Queue stalls without IRQ

Inspect producer/consumer indices, barriers, doorbell reachability, interrupt mask/status and firmware task state.

Corrupt outputs

Check tensor shape/stride, buffer bounds, cache sync, descriptor version, device address width and completion ordering.

Interrupt storm

Confirm source acknowledgement, level-versus-edge semantics, stale status, error retry loop and moderation policy.

Use-after-reset

Invalidate mappings, queue handles, fences and userspace contexts when the boot generation changes.

Teardown race

Stop new ioctls, wait/cancel in-flight work, synchronize IRQs and workqueues, then destroy mappings and device state.
