1. Databases & Storage

How data actually lives on disk and stays correct under concurrency and crashes: storage engine structures (B-trees, LSM-trees), the transaction machinery (ACID, isolation, MVCC, write-ahead logging), indexing strategy, and how databases scale out with replication and partitioning.

Key terms

ACID
Atomicity (all-or-nothing), Consistency (invariants hold), Isolation (concurrent transactions don't interfere), Durability (committed means survives a crash). The contract a transactional database offers.
Transaction
A group of reads and writes treated as one unit: either all of it commits or none of it does. The database's tool for multi-step invariants (debit one account, credit another).
B-tree
The classic storage/index structure: a wide, shallow, balanced tree of fixed-size pages. Reads and writes are O(log n) page accesses; updates happen in place. Default in Postgres, MySQL/InnoDB, SQL Server.
LSM-tree
Log-Structured Merge tree: writes buffer in a memtable, flush as sorted immutable files (SSTables), background compaction merges them. Fast sequential writes, reads may check several files. RocksDB, Cassandra, LevelDB.
WAL (write-ahead log)
Every change is appended to a sequential log before the data pages are touched. Commit = log flushed to disk. After a crash, replay the log to recover — durability from sequential I/O.
Index
A secondary structure (usually a B-tree) mapping column values to rows, turning full scans into seeks. Costs write amplification and space — every index must be maintained on each write.
Composite / covering index
Composite: an index over (a, b, c) — usable for prefixes (a), (a,b)… but not (b) alone. Covering: the index contains every column the query needs, so the table itself is never touched.
Selectivity
The fraction of rows a predicate matches. Indexes pay off on selective predicates (few rows); for low-selectivity ones (status = 'active' on 90% of rows) a scan is often cheaper.
Query planner
The optimizer that turns SQL into an execution plan using table statistics: which indexes, which join order, which join algorithm. EXPLAIN shows its choice — the first tool for slow queries.
Isolation levels
Read uncommitted → read committed → repeatable read → serializable: increasingly strict rules about what concurrent transactions can observe, trading anomaly prevention against concurrency.
Anomalies
Dirty read (see uncommitted data), non-repeatable read (row changes between two reads), phantom (new rows appear matching your predicate), lost update, write skew (two txns each read then write disjoint rows, jointly violating an invariant).
MVCC
Multi-Version Concurrency Control: writers create new row versions instead of overwriting; readers see a snapshot as of their start. Readers never block writers and vice versa. Postgres, InnoDB, Oracle.
Two-phase locking / deadlock
Lock-based isolation: acquire locks as you go, hold until commit. Two transactions locking rows in opposite orders deadlock; the database detects the cycle and kills one victim.
Normalization
Structuring tables so each fact is stored once (3NF and friends): no update anomalies, at the cost of joins. Denormalization copies data to kill joins — buy read speed with update complexity.
Replication
Copies of the data on multiple nodes: a leader takes writes and streams them to followers (read scaling, failover). Async replication risks losing the last writes on failover; sync costs latency.
Replication lag
The delay before a follower reflects the leader's writes. Read-your-own-writes violations ("I posted but it's gone") come from reading a lagging replica right after writing to the leader.
Sharding / partitioning
Splitting one dataset across nodes by a partition key (hash or range). Scales writes and size — but cross-shard queries and transactions become the application's problem. Hot keys skew load.
NoSQL families
Key-value (Redis, DynamoDB), document (MongoDB), wide-column (Cassandra), graph (Neo4j). Trade relational joins/transactions for a data model and scaling shape that fits the access pattern.
OLTP vs OLAP
Transaction processing: many small point reads/writes, row storage. Analytics: few huge scans/aggregations over billions of rows, column storage. Different engines because the access patterns are opposite.
Columnar storage
Store each column contiguously instead of each row: scans touch only needed columns, values compress extremely well (same type, similar values). ClickHouse, Parquet, BigQuery, Redshift.
Connection pooling
Database connections are expensive (memory, handshake, per-connection process in Postgres); a pool reuses a bounded set. Unbounded connections are a classic way applications take databases down.

Core concepts

B-tree vs LSM-tree

Two answers to "disks like sequential I/O". B-trees update pages in place: reads are predictable (one path root→leaf), writes cost random I/O and page splits. LSM-trees never update in place: writes are sequential appends (memtable → SSTable flush), so write throughput is superb; reads must consult memtable + several SSTables (bloom filters keep this sane), and background compaction consumes I/O — write amplification moves from foreground to background. Rule of thumb: read-heavy/OLTP → B-tree; write-heavy ingest (metrics, events, logs) → LSM. This is also why Cassandra ingests fast but point reads cost more than in Postgres.

WAL and crash recovery

Durability is not "write the data page" — that's random I/O and might be half-done at power loss. Instead: append the change to the WAL, fsync the log, acknowledge commit; the actual pages update later (checkpoints reconcile). Crash? Replay the log forward from the last checkpoint (redo), roll back losers (undo). Consequences worth knowing: commit latency ≈ one sequential fsync, group commit amortizes it across transactions, replication is usually just shipping the WAL, and turning off fsync trades durability for speed (fine for a cache, fatal for money).

Isolation levels as a menu

Serializable is "as if one at a time" — always correct, historically slow, now practical (Postgres SSI). Most databases default lower: read committed (Postgres) blocks dirty reads only — two reads in one transaction can disagree; repeatable read/snapshot (InnoDB default, Postgres RR) gives each transaction a frozen snapshot — but write skew slips through: two doctors both read "2 on call", each removes themselves, invariant broken, no conflict detected under snapshot isolation. Fixes at lower levels: SELECT … FOR UPDATE to lock what you read, unique/check constraints, or explicit serializable for the few transactions that need it.

Indexing strategy

An index earns its keep when it turns a scan of millions into a seek of dozens. Composite index column order matters: (tenant_id, created_at) serves "this tenant, recent first" perfectly and "all tenants yesterday" not at all — leftmost prefix rule. A covering index answers the query from the index alone (index-only scan). Every index taxes every INSERT/UPDATE and competes for cache. Functions on the indexed column (WHERE lower(email) = …) disable the index unless you index the expression. Workflow: find the slow query, EXPLAIN it, check what the planner sees, then add the one index that changes the plan.

Scaling: replicas, then shards

First move: leader-follower replication — reads fan out to replicas, writes stay on the leader. Costs: replication lag (stale reads; route read-your-writes traffic to the leader) and failover risk with async replication (unreplicated commits lost — a split brain if the old leader comes back). When writes or data size outgrow one leader: shard by a partition key. Choose the key by access pattern — hash of user_id spreads load but kills range queries; range keys keep scans but invite hot spots (today's date!). Cross-shard joins and transactions mostly stop existing; design so 99% of queries are single-shard.

Choosing a database

Default to a relational database until the access pattern says otherwise: transactions, joins, constraints, and 50 years of tooling are hard to beat. Reach for key-value when it's literally get/put by key at scale (sessions, carts); document when entities are self-contained aggregates read whole; wide-column for write-heavy time-series/event floods; search engines for text; columnar warehouses for analytics — often alongside the OLTP store, fed by CDC (change data capture), not instead of it. The expensive mistake is usually not the wrong engine but scattering one logical fact across engines with no source of truth.

2. OS, Concurrency & Runtime

What the machine gives your program: processes, threads, and virtual memory; how the kernel schedules and mediates everything through syscalls; the concurrency toolbox (locks, atomics, memory ordering) and its failure modes; and how CPU caches and async runtimes shape performance.

Key terms

Process
A running program with its own virtual address space, file descriptors, and at least one thread. Isolation boundary: one process cannot scribble on another's memory.
Thread
An execution stream inside a process: own stack and registers, shared heap and globals with its siblings. Cheap to communicate, easy to corrupt shared state.
Virtual memory
Each process sees a private, contiguous address space; the MMU maps virtual pages (typically 4 KB) to physical frames via page tables. Enables isolation, overcommit, swapping, and mmap.
Page fault
Access to a page with no valid mapping traps to the kernel. Minor: page is in RAM, just map it (cheap, how demand loading works). Major: fetch from disk (milliseconds — the performance cliff).
TLB
Translation Lookaside Buffer: a tiny cache of virtual→physical translations. Misses force a page-table walk; huge pages exist largely to relieve TLB pressure.
Syscall
The only door into the kernel: read, write, open, mmap, futex… A controlled mode switch (user→kernel), tens to hundreds of ns — cheap, but not free in a hot loop.
Context switch
The scheduler swaps which thread a CPU runs: save registers, switch stacks (and page tables across processes), warm caches go cold. Microseconds each — thousands per second is overhead you can measure.
Scheduler
Kernel component multiplexing runnable threads over CPUs, balancing fairness, latency (interactive tasks), and throughput. Preemptive: a timer interrupt can take the CPU away anytime.
Race condition
Outcome depends on unsynchronized timing of concurrent accesses (at least one write). Compiles fine, passes tests, corrupts data in production at 3 a.m.
Mutex
Mutual exclusion lock: one thread in the critical section at a time. Uncontended lock/unlock is nanoseconds (a CAS); contended means futex sleep — a syscall and context switch.
Semaphore
A counter with atomic wait (decrement-or-block) and signal (increment): bounds concurrent access to N (connection limits, bounded queues). A mutex is roughly a binary semaphore with ownership.
Condition variable
Wait (releasing a held mutex) until another thread signals a condition may now hold. Always re-check the predicate in a while loop — spurious wakeups are allowed.
Deadlock
Threads waiting on each other in a cycle, forever. Requires four conditions (mutual exclusion, hold-and-wait, no preemption, circular wait); break any one — most practically, a global lock order.
Atomic / CAS
Hardware-indivisible operations (compare-and-swap, fetch-and-add) on single words. The building block of every lock and lock-free structure. CAS loops retry under contention.
Memory ordering
Compilers and CPUs reorder memory accesses; other threads may observe writes out of program order. Atomics with acquire/release (or seq_cst) semantics insert the barriers that forbid it.
Cache line / coherence
CPUs move memory in 64-byte lines; MESI keeps per-core caches coherent — a write invalidates the line in every other core's cache. Coherence traffic is the hidden cost of sharing.
False sharing
Two threads write different variables that share a cache line: the line ping-pongs between cores as if they contended on one variable. Fix: pad/align hot per-thread data.
epoll / IO multiplexing
One thread waits on thousands of sockets and is told which are ready (epoll/kqueue/IOCP). The engine under nginx, Node, and every async runtime — replaces thread-per-connection.
Async runtime / event loop
Many logical tasks multiplexed on few threads: tasks yield at await points instead of blocking. Cheap concurrency for I/O-bound work; one CPU-hogging or blocking call stalls every task on that thread.
mmap
Map a file (or anonymous memory) into the address space: file I/O becomes memory access, pages loaded on fault, shared mappings enable zero-copy IPC. How executables and shared libraries load.
Garbage collection
Runtime reclaims unreachable heap objects (tracing: mark live from roots, sweep/compact the rest). Generational GCs exploit "most objects die young". Cost model: allocation rate and pause sensitivity.

Core concepts

Virtual memory earns its keep

One mechanism, many features: isolation (your pointer literally cannot name another process's memory), demand paging (start instantly, fault pages in as touched), copy-on-write (fork is cheap; both processes share pages until one writes), shared libraries (one libc in RAM, mapped everywhere), mmap-ed files, and overcommit. The costs surface as page faults — minor ones are routine; major faults mean disk, and a working set larger than RAM means thrashing: the machine is "busy" doing nothing but I/O. top shows it as high wait, low user CPU.

Threads vs async — pick by workload

Thread-per-request is simple and exploits all cores, but each thread costs stack memory and context switches; 10k blocked threads is real scheduler pressure. Async/event-loop flips it: thousands of I/O-bound tasks on a handful of threads, each parked task just a small state machine — this is how one box holds 100k connections. The catch: the loop must never block — one synchronous file read or CPU-heavy handler freezes every task behind it (offload to a worker pool). CPU-bound work wants threads ≈ cores and no async ceremony at all. Most servers end hybrid: event loop for I/O, pool for crunching.

The data race, precisely

Two threads access the same location, at least one writes, no synchronization orders them — that's a data race, and in C/C++/Go it's undefined behavior, not merely nondeterminism: the compiler assumed no race when optimizing. count++ is three operations; two threads interleave and increments vanish. Fixes ordered by preference: don't share (per-thread data, message passing); share immutably; share behind a mutex; atomics for counters and flags. Tools catch what review misses: ThreadSanitizer and Go's -race find real interleavings in tests — run them in CI.

Locks that don't hurt

Granularity is the game: one big lock is correct but serializes everything (Python's GIL); fine-grained locks scale but invite deadlock and subtle ordering bugs. Rules that survive contact with production: keep critical sections tiny (never do I/O holding a lock), acquire multiple locks in one global order everywhere, prefer reader-writer locks only when reads truly dominate (they're costlier than they look), and measure contention before "optimizing" it. When a counter is the hot spot, an atomic beats a mutex; when a structure is, sharding it (N locks by key hash) usually beats lock-free heroics.

Memory ordering exists

The classic trap: thread A writes data = 42; ready = true; thread B spins on ready then reads data — and sees stale data, because compiler or CPU reordered the writes, or hoisted the read of a plain ready out of the loop entirely. Plain loads/stores promise nothing across threads. Release store on ready + acquire load pairs them: everything written before the release is visible after the acquire. That release/acquire pair is exactly what mutex unlock/lock gives you — which is why "just use a lock" is correct advice, and why lock-free code is expert territory.

Mechanical sympathy

Latency ladder, roughly: L1 ~1 ns, L2 ~4 ns, L3 ~a dozen, RAM ~100 ns, SSD ~100 µs, disk seek ~ms, same-DC network ~500 µs. Two consequences dominate real profiles: locality (arrays beat pointer-chasing linked structures because the prefetcher can see the future) and sharing costs (every cross-core write to a shared line pays coherence traffic — false sharing makes "independent" counters contend). Branch mispredicts flush pipelines; syscalls and context switches trash caches. Optimizing = measuring first: perf, flame graphs, and cache-miss counters, not intuition.

3. Distributed Systems

What changes when the system spans machines that fail independently and a network that loses, delays, and duplicates messages: consistency models and CAP, consensus and quorums, delivery guarantees and idempotency, distributed transactions, and the defensive patterns — timeouts, retries with jitter, circuit breakers, fencing tokens.

Key terms

Partial failure
The defining property: some nodes/links fail while others work, and you often cannot tell "crashed" from "slow" from "unreachable". A timeout is a suspicion, not a fact.
CAP theorem
During a network Partition you must choose: refuse some requests (Consistency) or serve possibly-stale data (Availability). No partition, no forced choice — see PACELC for the latency tradeoff the rest of the time.
Linearizability
The strong one: every operation appears to take effect atomically at some instant between its start and end; after a write completes, all reads see it. What single-node RAM gives you; expensive across nodes.
Eventual consistency
Replicas converge if writes stop; meanwhile reads may be stale or out of order. Fine for likes and carts; dangerous for uniqueness, balances, and anything with "exactly" in the requirement.
Causal consistency
Middle ground: effects are seen after their causes (reply never appears before the post it answers), without paying for a global order of everything.
Consensus (Raft/Paxos)
Protocol for nodes to agree on a value/log despite crashes: elect a leader, replicate entries, commit when a majority has them. Raft is the understandable one — etcd, Consul, CockroachDB.
Quorum
Majority (or R/W-configurable) subset that must participate. With N replicas, W + R > N makes read and write sets overlap, so a read meets the newest write. Majorities also prevent two leaders.
Split brain
Two halves of a partitioned cluster each believe they're in charge and accept conflicting writes. Prevented by majority quorums — the minority side must stop serving.
Fencing token
Monotonic number issued with a lock/lease; storage rejects writes with an older token. Defends against the zombie: a paused ex-leader waking up and writing as if it still held the lease.
Replication: sync vs async
Sync: leader waits for follower ack — durable but slow, stalls if the follower dies. Async: fast, but failover loses the tail of unreplicated writes. Semi-sync (wait for one of K) is the common compromise.
Idempotency
Applying an operation twice equals applying it once (PUT x=5 yes; "add $100" no). The only practical route to "effectively exactly once": at-least-once delivery + idempotent handlers (dedupe keys).
Delivery guarantees
At-most-once (may drop, never duplicates), at-least-once (never drops, may duplicate — the practical default), exactly-once (end-to-end, only as at-least-once + dedupe/transactions).
Two-phase commit (2PC)
Coordinator asks all participants to prepare (vote), then broadcasts commit/abort. Atomic across systems — but blocking: participants hold locks while a crashed coordinator is down.
Saga
Long transaction as a chain of local transactions, each with a compensating action (refund, release, cancel) run in reverse on failure. Trades atomicity for availability; intermediate states are visible.
Outbox pattern / CDC
Write the business row and an "event to publish" row in one local transaction; a relay publishes from the outbox. Kills the "DB committed but message never sent" dual-write bug.
Logical / vector clocks
Physical clocks skew, so order by causality instead: Lamport clocks give a consistent order; vector clocks additionally detect that two updates were concurrent (a real conflict).
CRDT
Conflict-free Replicated Data Type: state merges commutatively, so concurrent updates converge without coordination (G-counter, OR-set). Collaborative editors, offline-first apps.
Retry + backoff + jitter
Retry transient failures with exponentially growing waits and randomization. No jitter → synchronized retry stampedes. Only retry idempotent operations blindly.
Circuit breaker
After repeated failures to a dependency, fail fast for a cooldown instead of stacking timeouts — protects your threads and the struggling service; half-open probes test recovery.
Backpressure
Slowing producers when consumers lag (bounded queues, credit windows) instead of buffering forever. Unbounded queues turn overload into memory exhaustion plus latency you can't recover from.

Core concepts

The network is the problem

Every distributed-systems pattern is downstream of one fact: messages can be lost, delayed, reordered, or duplicated, and a non-answer doesn't tell you which. Send a request, get silence — did it fail before processing, or after? You cannot know; you can only time out and decide what to do with the ambiguity. That's why retries exist (and demand idempotency), why "did the payment go through" is a design problem, and why leases and fencing tokens replace "I'm sure it's dead" with mechanisms that stay safe when you're wrong. Design rule: every remote call has a timeout, a retry policy, and an answer to "what if it executed twice?"

CAP without the folklore

CAP is narrower than its reputation: only during a partition must you pick consistency (some requests error/wait) or availability (serve stale). The everyday tradeoff is PACELC's other half — Else, latency vs consistency: a linearizable read must coordinate (quorum round trip); a local replica read is fast and possibly stale. Pick per operation, not per system: account balance and username uniqueness want CP paths; feeds, counts, and sessions tolerate staleness happily. Systems advertise this as read/write concern levels (Dynamo-style R/W, Mongo readConcern, etc.) — the knobs are the theorem made concrete.

Raft in one breath

Nodes are followers until heartbeats stop; a timed-out follower becomes candidate, requests votes, and a majority makes it leader for that term. Clients talk to the leader; it appends entries to its log, replicates, and commits once a majority holds them — then followers apply. Safety comes from majorities overlapping: any new leader's electorate contains a node with every committed entry, and voters refuse candidates with stale logs. Terms fence off zombies (higher term wins, old leader steps down). A 5-node cluster survives 2 failures; the minority side of a partition can't elect anyone or commit anything — that's the split-brain proof.

Exactly-once is a team effort

The broker can promise at-least-once; "exactly once" is something you build end to end. Producer side: outbox pattern so the DB write and the event publish can't disagree. Consumer side: dedupe on an idempotency key (store processed IDs transactionally with the side effect), or make the operation naturally idempotent (set, not increment). The same discipline covers API design: clients send an idempotency key with POSTs, the server stores the first response and replays it on retry — Stripe-style. If a handler isn't idempotent and isn't deduped, at-least-once delivery will eventually double-charge someone.

Distributed transactions, reluctantly

Cross-service atomicity is expensive in every currency. 2PC gives real atomicity but blocks: prepared participants hold locks while a dead coordinator is down, and availability multiplies down across participants — which is why it rarely crosses trust or service boundaries. Sagas accept non-atomicity: local commits plus compensations, with visible intermediate states and business-level design for "booked then unbooked". Often the honest fix is upstream: redraw service boundaries so the invariant lives inside one database, where a plain transaction handles it. The best distributed transaction is the one you refactored away.

Defensive patterns that compose

Overload cascades are the classic outage shape: one slow dependency, callers pile up retries, thread pools drain, everything upstream browns out. The standard kit, layered: timeouts on everything (deadline propagation so the whole chain shares one budget), exponential backoff with jitter (spread the herd), retry budgets (retries as a % of traffic, not per-request heroics), circuit breakers (fail fast when it's clearly down), bounded queues with load shedding (reject early beats collapse later), and bulkheads (per-dependency pools so one bad neighbor can't take the house). Each is simple; reliability is their composition.

4. Networking

The path a request actually takes: DNS, TCP's reliability machinery and congestion control, TLS 1.3, and the HTTP generations' fight against head-of-line blocking — plus the operational layer of load balancers, NAT, and tail latency.

Key terms

Layered model
Each layer uses the one below and serves the one above: link (Ethernet/WiFi) → network (IP, best-effort packets) → transport (TCP/UDP, end-to-end) → application (HTTP, DNS, TLS in between). Encapsulation: each layer wraps the payload with its header.
IP
Best-effort packet delivery between addresses across networks: packets may be lost, reordered, duplicated. Routers forward hop by hop by destination prefix. Everything reliable is built on top.
TCP
Reliable, ordered byte stream over IP: sequence numbers, acks, retransmission, flow control, congestion control. Connection-oriented — the 3-way handshake establishes state on both ends.
UDP
Just IP plus ports: no connection, no ordering, no retransmission. For when the application wants control (DNS queries, QUIC, games, video) or can't afford TCP's machinery.
3-way handshake
SYN → SYN-ACK → ACK: exchanges initial sequence numbers and establishes the connection. One RTT before any data — why connection reuse and 0-RTT tricks matter.
Flow control
Receiver advertises a window ("I can buffer this much"); sender never exceeds it. Protects the receiver. Distinct from congestion control, which protects the network.
Congestion control
Sender probes network capacity: slow start (exponential growth) until loss/ECN, then additive increase/multiplicative decrease around a congestion window. CUBIC and BBR are the modern algorithms.
Head-of-line blocking
One lost/slow item stalls everything queued behind it: a lost TCP segment blocks delivery of all later bytes (kernel holds them until the gap fills), one slow HTTP/1.1 response blocks the pipeline.
DNS
The hierarchy that turns names into records: stub → recursive resolver → root → TLD → authoritative, with caching by TTL at every step. A / AAAA (addresses), CNAME (alias), MX, NS, TXT.
TTL / caching
Each DNS record carries a time-to-live; resolvers serve from cache until expiry. Low TTL = fast failover, more queries; high TTL = efficient, slow to change. "DNS propagation" is just caches expiring.
TLS
Encryption + integrity + authentication above TCP. The handshake negotiates keys (ECDHE — ephemeral, giving forward secrecy) and authenticates the server by certificate chain to a trusted CA. TLS 1.3: 1 RTT.
Certificates / PKI
A certificate binds a name to a public key, signed by a CA the client already trusts (chain to a root store). Wrong name, expired, or untrusted chain = the browser warning everyone clicks through in dev.
HTTP/1.1
Text protocol, one request at a time per connection (keep-alive reuses it). Browsers open ~6 parallel connections per host to fake concurrency — request-level HOL blocking otherwise.
HTTP/2
Binary framing multiplexes many streams over one TCP connection, plus header compression (HPACK). Kills request-level HOL — but one lost TCP packet still stalls all streams (transport-level HOL remains).
HTTP/3 / QUIC
Transport over UDP with streams that are independent at the transport level (a lost packet stalls only its stream), TLS 1.3 built in, 0/1-RTT setup, and connection migration across IP changes.
WebSocket / gRPC
WebSocket: upgrade an HTTP connection to full-duplex messaging (live updates, chat). gRPC: RPC over HTTP/2 with protobuf contracts and streaming — the typed service-to-service default.
Load balancer L4 vs L7
L4 spreads TCP connections by IP/port — fast, protocol-blind. L7 terminates HTTP(S): routes by path/host/header, retries, sticky sessions, TLS offload — smarter, costlier. Health checks eject dead backends.
NAT
Router rewrites private addresses/ports to a public one, keeping a translation table keyed by connection. Why "the same IP" serves a whole office, and why inbound connections need port forwarding or hole punching.
MTU
Maximum frame payload (~1500 bytes on Ethernet). Bigger IP packets fragment (or are dropped when DF is set — path MTU discovery relies on ICMP that firewalls love to block: silent black holes).
Latency vs bandwidth
Bandwidth buys throughput; the speed of light sets latency. A cross-ocean RTT is ~150 ms no matter the pipe — so protocols fight round trips (fewer handshakes, pipelining, caching, CDNs).
Tail latency
p99 matters more than mean: fan out to 100 backends and a 1%-slow straggler hits ~63% of requests. Fought with hedged requests, timeouts, and keeping queues short.

Core concepts

URL to pixels — the whole path

Type a URL: DNS resolves (cache → recursive resolver → root/TLD/authoritative), TCP handshakes with the answer (1 RTT), TLS 1.3 handshakes (1 more RTT), the HTTP request goes out, a CDN edge or load balancer picks a backend, the response streams back, and the browser parses/renders as bytes arrive. Count round trips: on a 150 ms RTT path, cold connection setup alone is ~300 ms before byte one. That arithmetic explains most web performance engineering: connection reuse, session resumption/0-RTT, CDNs moving content near users, and QUIC folding transport+TLS into one round trip.

How TCP makes IP reliable

Every byte is numbered; the receiver acks the highest in-order byte received; the sender retransmits on timeout or on duplicate acks/SACK (fast retransmit). Delivery to the app is strictly in order — a lost segment means later segments wait in the kernel buffer until the hole fills (that's transport HOL blocking). Two independent throttles govern sending: the receiver's advertised window (flow control) and the congestion window (network protection); the effective window is the minimum. Teardown: FIN/ACK both ways, with TIME_WAIT lingering to absorb stragglers — why servers churning short connections exhaust ports.

Congestion control intuition

The network gives no capacity signal, so TCP probes: slow start doubles the congestion window every RTT until loss (or ECN mark), then backs off multiplicatively and settles into cautious additive growth — sawtooth around the fair share. Modern twists: CUBIC (default) grows by a cubic curve for fat pipes; BBR models bottleneck bandwidth and RTT instead of waiting for loss — better on lossy/long paths. Operational consequences: throughput ≈ window/RTT (long-RTT links need big windows), a cold connection is slow by design (slow start), and bufferbloat — oversized router buffers — turns "more buffering" into seconds of queueing latency.

TLS 1.3 — what got fixed

One round trip: ClientHello carries the key share (ECDHE) up front; the server answers with its share, certificate, and proof of key possession; both derive session keys and the client's first data rides the very next flight. Resumption can even send 0-RTT data (replayable — idempotent requests only). Forward secrecy is default: session keys come from ephemeral exchanges, so a later key compromise can't decrypt recorded traffic. Legacy weaknesses (static RSA, weak ciphers, renegotiation) were removed outright. Authentication still hinges on the certificate chain — and most real-world TLS failures are still expired or misnamed certs.

The HOL-blocking arms race

HTTP/1.1: one response at a time per connection — request-level HOL; browsers work around it with ~6 connections and devs with sprites/bundling. HTTP/2: multiplex streams over one connection — request-level HOL solved, but TCP's in-order bytestream means one lost packet stalls every stream (worse on lossy links than 6 separate connections!). HTTP/3/QUIC: streams are independent transport-level units over UDP — loss stalls only the affected stream. The pattern to remember: each generation moved the same bottleneck down a layer until the transport itself had to be replaced.

Load balancing and tails

L4 balancers push packets — millions of connections, no protocol smarts; L7 proxies terminate HTTP and route by content, retry failed backends, offload TLS, and emit the metrics you'll actually debug with. Algorithms matter under load: round-robin ignores slow backends; least-connections adapts; consistent hashing keeps caches warm and shuffles little when nodes churn. And tails dominate SLOs: if p99 = 500 ms and a page fans out to 50 services, most pages eat a p99 somewhere. Countermeasures — hedged requests after a delay, strict deadlines, load shedding — buy more UX than raising the mean ever will.

5. Infrastructure

Running software as a fleet: containers and orchestration, infrastructure as code, CI/CD and safe rollout strategies, the observability stack that tells you what production is doing, SLOs that decide when to stop shipping, and the caching and queueing layers every architecture grows.

Key terms

VM vs container
VM: hardware virtualized, each guest runs its own kernel — strong isolation, minutes to boot, GBs. Container: processes sharing the host kernel, isolated by namespaces + cgroups — starts in ms, MBs. Different trust boundaries.
Namespaces / cgroups
The two kernel features under containers: namespaces give a private view (PIDs, network, mounts, users); cgroups cap resources (CPU, memory — exceed the memory limit and the OOM killer visits).
Image / layers
A container image is a stack of immutable filesystem layers plus config. Layers are cached and shared: order your Dockerfile so rarely-changing steps (deps) precede frequently-changing ones (source) for fast rebuilds.
Registry / tags
Images live in registries, pulled by tag or digest. Tags are mutable pointers — :latest today isn't :latest tomorrow; deploy by digest or immutable version tags for reproducibility.
Kubernetes
Declarative container orchestrator: you write desired state, controllers reconcile reality toward it — scheduling, restarts, scaling, rollouts. The API server + etcd hold the cluster's truth.
Pod
Smallest deployable unit: one or more tightly-coupled containers sharing network namespace (localhost) and volumes. Pods are cattle — they die and are replaced, never repaired.
Deployment / ReplicaSet
Deployment declares "N replicas of this pod template" and manages rolling updates between versions; the ReplicaSet keeps the replica count true at each version.
Service / Ingress
Service: stable virtual IP/DNS name load-balancing over healthy pods selected by label (pod IPs churn constantly). Ingress/Gateway: HTTP routing from outside — host/path → Service, TLS termination.
Liveness / readiness probes
Liveness: "restart me if this fails" (hung process). Readiness: "don't route traffic to me yet" (warming up, draining). Confusing them causes restart loops or black-holed requests.
Requests / limits
Request: what the scheduler reserves for the pod. Limit: hard cap (CPU throttled at limit; memory OOM-killed). The gap between them is overcommit — and your noisy-neighbor exposure.
IaC (Terraform etc.)
Infrastructure declared in versioned code, applied by a tool that diffs desired vs actual state (plan → apply). Reviewable, repeatable, revertable — the opposite of console click-ops and snowflake servers.
CI/CD
CI: every push builds and tests automatically. CD: artifacts flow to environments through an automated, gated pipeline. Small frequent deploys beat big-bang releases on every reliability metric.
Blue-green
Two full environments; switch traffic atomically to the new one and keep the old warm for instant rollback. Simple, fast to revert, doubles infrastructure during the swap.
Canary
Route a small slice (1–5%) of real traffic to the new version, compare error/latency metrics against the fleet, then ramp or roll back. Limits blast radius; needs good metrics to judge the verdict.
Feature flag
Deploy code dark, enable per user/percent at runtime, kill instantly without a deploy. Decouples deploy from release; flags need cleanup or they become permanent config debt.
Logs / metrics / traces
The three signals: structured logs (what happened, searchable), metrics (cheap numeric time series — rates, latencies, saturation — what you alert on), traces (one request's path across services with per-hop timing).
SLI / SLO / error budget
SLI: measured indicator (p99 latency, success rate). SLO: target (99.9% monthly). Error budget: the allowed 0.1% — spend it shipping fast; exhausted budget pauses features for reliability work.
Autoscaling
Horizontal: add/remove replicas on load (needs stateless-ish services). Vertical: resize instances. Reactive scaling lags spikes — keep headroom; scale on leading indicators (queue depth, RPS) not just CPU.
Message queue / pub-sub
Queue (SQS, RabbitMQ): each message consumed by one worker — decouple, absorb bursts, retry with DLQ. Log/pub-sub (Kafka): append-only, many independent consumer groups replay at their own pace.
Cache (Redis / CDN)
Serve hot data from memory/edge instead of recomputing: cache-aside with TTLs is the workhorse. The hard parts: invalidation, stampedes on hot-key expiry, and cold-cache thundering herds after deploys.
Secrets management
Credentials in a vault (Vault, cloud secret managers), injected at runtime, rotated, audited — never in git, images, or env files. Prefer short-lived credentials and workload identity over static keys.

Core concepts

Containers are processes

A container is not a lightweight VM — it's a normal Linux process wearing namespaces (private PID/network/mount view) and cgroups (resource caps), rooted in an image's layered filesystem. That's why it starts in milliseconds, why all containers on a host share one kernel (kernel exploit = shared blast radius; use VMs/gVisor across hostile tenants), and why "works in the container" is mostly "the filesystem is pinned". Images build reproducibly in layers — order Dockerfile steps by change frequency, keep images minimal (distroless: smaller pulls, smaller CVE surface), and never bake secrets into a layer: layers are forever.

Kubernetes = reconciliation

Everything in Kubernetes is one idea: desired state lives in etcd, controllers watch it and nudge reality toward it, forever. You don't "run a pod" — you declare a Deployment of 3 replicas; the controller notices 2 exist and starts one, notices a node died and reschedules, notices a new image and rolls pods gradually behind a Service that only routes to ready ones. Self-healing falls out of the loop, not from any special feature. The operational corollary: never hand-edit live objects (the controller reverts you, or worse, GitOps drifts) — change the declaration, let the loop do the work.

Rollouts that fail safely

Assume some deploys are bad; choose how you find out. Rolling update: gradual, cheap, but old and new coexist — every schema and API change must be backward compatible for the overlap. Blue-green: atomic cutover, instant rollback, double cost. Canary: a metric-judged slice of real traffic — the best failure detector, if your dashboards can actually tell good from bad. Underneath all three: expand-contract for databases (add column → dual-write/migrate → cut over → drop later; never break the running version) and feature flags so risky code ships dark and turns off faster than any rollback.

Observability is for unknowns

Monitoring answers "is it broken?" — observability answers "why?", including failures you never predicted. Metrics are the cheap always-on layer: RED (rate, errors, duration) per service, USE (utilization, saturation, errors) per resource; alert on symptoms users feel (SLO burn rate), not on every twitching cause. Traces stitch one request across twenty services and point at the slow hop; logs carry the detail, joined to traces by correlation IDs. The discipline that makes it work: instrument at the edges, propagate context, sample intelligently — and delete alerts nobody acts on, because a noisy pager is an untrusted pager.

SLOs make reliability a number

Pick an SLI users feel (successful requests under 300 ms), set a target you can defend (99.9%), and the complement becomes an error budget: ~43 minutes of violation per month. Now decisions stop being vibes: budget left → ship features; budget burned → freeze launches and fix reliability, by prior agreement with product. Alerts follow burn rate — "at this pace the month's budget dies in 6 hours" pages someone; a slow leak makes a ticket. 100% is not a target: the last nine costs more than users notice, and a small budget deliberately spent (chaos drills, risky migrations) buys learning.

Queues and caches, honestly

A queue decouples: producers absorb bursts, consumers scale on backlog, retries with dead-letter queues contain poison messages — at the price of eventual consistency and duplicate delivery (consumers must be idempotent; monitor queue age, not just depth). A cache buys latency: cache-aside with TTL serves most needs, but plan for the failure modes — stampede when a hot key expires (locking or early refresh), inconsistency windows after writes (invalidate, don't update-in-place), and the cold-start herd after a flush. Both share a law: they hide load problems until the day they amplify them — capacity-plan for the cache-miss path, not the hit path.
